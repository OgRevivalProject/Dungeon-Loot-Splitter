🧙 Logic Explanation

Array as Source of Truth: All loot lives in a single array called loot declared at the top of script.js. The screen never stores values — it reflects only what the array contains. When loot is added, removed, or party size changes, the array updates first, then updateUI() recalculates everything from that array. The display follows the data; the data never follows the display.

Quantity in the Data Model: Each loot object carries name, value, and quantity. Storing quantity in the object means the data model is complete and self-contained. Total value per item is computed as value × quantity inside the loop in updateUI(), ensuring the math always reflects the actual object — not a stale UI value.

updateUI() as the Single Renderer: All calculations and all DOM writes happen inside updateUI(). No other function calculates totals or updates visible output. Every state-changing action — add, remove, party size change, split, reset — calls updateUI() immediately after mutating state. This eliminates duplicate logic and guarantees the interface is never out of sync.

State Controls Visibility and Actions: updateUI() uses conditional logic to show or hide sections and enable or disable buttons. The loot table is hidden when the array is empty. The results panel is hidden when party size is invalid or the loot array is empty. The Split button is disabled until both conditions are valid. Visibility is controlled exclusively by adding or removing the .hidden CSS class — JavaScript never touches style properties directly.

Preventing Invalid State: Validation runs inside addLoot() before any mutation. Name must be non-empty, value must be a valid non-negative number, and quantity must be at least 1. If any check fails, the array is not touched and an error displays in the UI. The same rules apply during restoration — each saved item is re-validated before being pushed back into the live array.

How State Is Saved and Restored: saveState() packages the current loot array and partySize into a single object and writes it to localStorage using JSON.stringify(). On page load, restoreState() retrieves that string, parses it back into an object with JSON.parse(), validates every field, and assigns the results to the live variables before updateUI() runs. This means the app wakes up in exactly the state it was left in. JSON.stringify() converts a JavaScript value into a plain text string so it can be stored. JSON.parse() converts that string back into a live JavaScript object.

Why updateUI() Must Run After Restoration: Restoring state only puts the data back into memory — it doesn't touch the DOM. The page still shows the default empty state until updateUI() runs and rebuilds the loot list, recalculates totals, and re-enables the split button based on what was restored. Without that call, the data would be there but invisible.

Debugging with Breakpoints: Setting a breakpoint inside restoreState() just after JSON.parse() and before the validation loop makes it possible to inspect the parsed object in the browser's Variables panel. Stepping forward confirms each item passes validation before entering the array, and confirms that updateUI() doesn't run until restoration is complete. This makes the lifecycle order — restore, then render — visible and verifiable rather than assumed.

The Reset Lifecycle: Reset is not just cosmetic. Clearing the array and setting partySize back to 1 resets in-memory state. Calling localStorage.removeItem() destroys the persisted state so a future page load starts fresh. Then updateUI() reflects that empty state immediately. Without removing from localStorage, the data would come back on the next refresh even after a reset.
