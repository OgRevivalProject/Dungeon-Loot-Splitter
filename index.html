🧙 Logic Explanation

Array as Source of Truth: All loot lives in a single array called loot declared at the top of script.js. The screen never stores values — it reflects only what the array contains. When loot is added, removed, or party size changes, the array updates first, then updateUI() recalculates everything from that array. The display follows the data; the data never follows the display.

Quantity in the Data Model: Each loot object now carries name, value, and quantity. Storing quantity in the object means the data model is complete and self-contained. Total value per item is computed as value × quantity inside the loop in updateUI(), ensuring the math always reflects the actual object — not a stale UI value.

updateUI() as the Single Renderer: All calculations and all DOM writes happen inside updateUI(). No other function calculates totals or updates visible output. Every state-changing action — add, remove, party size change, split — calls updateUI() immediately after mutating state. This eliminates duplicate logic and guarantees the interface is never out of sync.

State Controls Visibility and Actions: updateUI() uses conditional logic to show or hide sections and enable or disable buttons. The loot table is hidden when the array is empty. The results panel is hidden when party size is invalid or the loot array is empty. The Split button is disabled until both conditions are valid. Visibility is controlled exclusively by adding or removing the .hidden CSS class — JavaScript never touches style properties directly.

Preventing Invalid State: Validation runs inside addLoot() before any mutation. Name must be non-empty, value must be a valid non-negative number, and quantity must be at least 1. If any check fails, the array is not touched and an error displays in the UI. Party size is validated inside updateUI() before the per-member split is calculated — invalid party size prevents recalculation entirely.
